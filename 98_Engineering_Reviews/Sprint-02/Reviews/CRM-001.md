## Assumptions

- Tag assignment accepts string array on create/update; tags are find-or-created per organization (no tag management UI).
- Notes are owner/manager only; receptionist can create contacts but not add notes or edit/delete.
- Marketing and accountant read access enforced via role codes in `ContactPolicy`, not RBAC permission seeds alone.
- Pagination uses offset/limit (first pagination in codebase; no Kaminari/Pagy gem).
- `contact_tag_assignments` uses UUID primary key (not composite) for Rails compatibility.
- User foreign keys use UUID type to match existing `users` table.

---

## Suggestions

- Add `customers.view` permission to marketing/accountant RBAC seeds for consistency with future permission-based UI gates.
- Consider counter cache on `contact_notes` instead of manual `notes_count` updates.
- Add dedicated notes list endpoint if note history UI is needed beyond timeline events.

---

## Risks

- ILIKE search on large contact volumes may need full-text indexing later.
- Filter combinations (e.g. recently_contacted + status) are independent; no compound filter validation.

---

## Technical Debt

- No dedicated command/query specs (covered by request specs).
- Frontend create/edit routes are not separately guarded beyond API 403 responses.

---

## Performance Notes

- List/search queries eager-load `contact_tags`.
- Dashboard runs multiple count queries; acceptable at foundation scale.

---

## Security Notes

- All endpoints require authentication; organization scoping on every query.
- Soft delete prevents data loss; deleted contacts excluded from kept scope.
- Role-based authorization on all mutating actions.
