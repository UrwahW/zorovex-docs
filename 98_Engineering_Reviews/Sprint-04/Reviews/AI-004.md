## Summary

Implemented AI-004 per `09_Cursor/Sprint_01/AI/AI-004.md`. This introduces the provider-independent Zorovex AI Runtime — the execution engine for all AI agents. The runtime loads agent configuration and organization context, resolves knowledge and goals, registers tools (without executing them), compiles prompts, executes via **MockProvider only**, inspects output, estimates tokens/cost, and persists structured runtime logs.

OpenAI, Anthropic, and Gemini adapters are registered behind `ProviderInterface` but raise `ProviderNotConfiguredError` if selected. Tool registry registers Booking, CRM, Business, Communication, Report, and Inventory tools with simulated action metadata only.

Frontend adds an AI Studio **Runtime** page with dashboard, inspector, execution timeline, provider selector, knowledge/goals/tools viewers, and execution log. Access matches spec: owners execute; owners and managers read logs/providers.

**Verification:** 632 RSpec examples (0 failures), 86 Vitest tests (0 failures), Brakeman 0 warnings, RuboCop clean on AI runtime paths.

## Runtime Architecture

```
Message
  → ContextBuilder (org, agent, business, platform)
  → KnowledgeResolver (enabled sources only)
  → GoalResolver (config + keyword triggers + escalations)
  → ToolRegistry.select_for_goals (register only)
  → PromptCompiler (sole prompt generator)
  → ProviderRegistry → MockProvider.execute
  → TokenEstimator + CostEstimator
  → RuntimeInspector
  → RuntimeLogger (stdout events + DB persist)
```

### Provider Abstraction

| Provider | Executes | Health |
|----------|----------|--------|
| Mock | Yes | `ok` |
| OpenAI | No | `not_configured` |
| Anthropic | No | `not_configured` |
| Gemini | No | `not_configured` |

### Registered Tools (no execution)

Booking, CRM, Business, Communication, Report, Inventory — simulated `booking_lookup`, `crm_lookup`, `route_conversation` actions when selected.

### Events Logged

`RuntimeExecuted`, `PromptCompiled`, `KnowledgeResolved`, `ProviderExecuted`, `RuntimeFailed`

## Assumptions

1. **MockProvider** generates deterministic heuristic responses from resolved goals/knowledge (not LLM output).
2. **Goal labels** follow AI-004 spec (`Book Appointment`, `Upsell`, etc.) mapped from receptionist configuration goal keys.
3. **Knowledge sources** beyond receptionist catalog (`packages`, `pricing`, `crm`, `bookings`) load only when present in agent `knowledge_sources` configuration.
4. **First agent** in the organization is used on the Runtime dashboard execute form (single-agent UX for now).
5. **Platform gating** reuses existing `ai_runtime` feature from PLATFORM-001.
6. **Promotions** knowledge maps to FAQ-type business knowledge items (consistent with receptionist catalog).

## Risks

| Risk | Mitigation |
|------|------------|
| Mock responses may diverge from future real providers | Provider interface + inspector designed for adapter swap |
| Goal keyword matching is heuristic | Goals also driven by agent configuration toggles |
| Extended knowledge keys not in receptionist validator | Runtime loads only configured keys; validation unchanged |
| `runtime_errors` column naming | Renamed from `errors` to avoid ActiveRecord conflict |
