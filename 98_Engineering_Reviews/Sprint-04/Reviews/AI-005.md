## Summary

Implemented AI-005 per `09_Cursor/Sprint_01/AI/AI-005.md`. The AI Runtime now detects intent, selects a workflow, and hands execution to the **Workflow Engine** — business logic lives in workflows and `BusinessActions`, not in the runtime. Seven workflows are registered: Booking, Lead, FAQ, Complaint, Cancellation, Escalation, and Recommendation.

The **Booking Workflow** implements the full state machine (service → date → time → availability → payment → human verification → booking → confirmation) with deposit rules from `BusinessPaymentSetting`, ICS calendar export, CRM/comms integration via existing booking services, and human handoff. Workflow sessions persist state/context, support resume/assign/cancel, and log every transition.

AI Runtime invokes workflows only when `ai_actions` is enabled **and** a `contact_id` or `thread_id` is provided (conversation-driven). Otherwise the mock provider path from AI-004 remains unchanged.

## Workflow Engine

```
Intent (goals + message)
  → WorkflowRegistry
  → WorkflowSession (new or resumed)
  → WorkflowStateMachine
  → BusinessActions (bookings, CRM, comms)
  → WorkflowEvents log
  → Comms thread message (system)
```

### Registered Workflows

| Key | Label |
|-----|-------|
| `booking` | Booking Workflow |
| `lead` | Lead Workflow |
| `faq` | FAQ Workflow |
| `complaint` | Complaint Workflow |
| `cancellation` | Cancellation Workflow |
| `escalation` | Escalation Workflow |
| `recommendation` | Recommendation Workflow |

### Booking State Machine

`collecting_service` → `collecting_date` → `collecting_time` → `checking_availability` → `collecting_payment` → `waiting_payment` → `waiting_human` → `creating_booking` → `confirmed`

### Business Actions (no logic in AI Runtime)

- `Bookings::BookingService#create` (source: `future_ai`)
- `Bookings::CancelService` / `RescheduleService`
- `Crm::Contacts::CreateCommand` (leads)
- Contact notes + activity logging
- `Comms::ConversationService` (thread escalation, system messages)

### Payment

Reads `BusinessPaymentSetting` — supports cash, JazzCash, EasyPaisa, bank transfer, Raast, custom link. No payment gateways. Manual verification via `POST /workflows/:id/resume` with `resume_action: verify_payment`.

### ICS Export

`Workflows::IcsGenerator` produces downloadable `.ics` for confirmed bookings via `GET /api/v1/workflows/:id/ics`.

| Method | Path | Access |
|--------|------|--------|
| GET | `/api/v1/workflows` | Owner, Manager, Receptionist, Marketing, Accountant |

Runtime execute accepts optional `contact_id` and `thread_id` for workflow continuity.

## Assumptions

1. Workflows require a conversation context (`contact_id` or `thread_id`) when invoked from AI Runtime.
2. Payment methods and deposit rules are configured per organization via `business_payment_settings` (defaults created on first access).
3. Booking availability uses existing `Bookings::AvailabilityService` / `Operations::AvailabilityEngine`.
4. Message parsing for dates/times/services is heuristic (not NLP); sufficient for AI-005 acceptance.
5. `ai_actions` platform feature gates workflow execution (enterprise/custom plans).
6. ICS is generated inline in session context — no external calendar sync.
