## Summary

Four architectural refinements were applied to the existing Receptionist Configuration implementation. No new features were added. Changes prepare the configuration JSON for future schema migrations, audit metadata, and clearer field naming for AI Runtime evolution.

**Verification:** 470 RSpec examples (0 failures), 57 Vitest tests (0 failures), RuboCop clean on changed files, Brakeman 0 warnings.

---

## 1. Configuration Schema Version

Every agent configuration now includes:

```json
{
  "metadata": {
    "schema_version": 1
  }
}
```

- `ReceptionistConfiguration::Metadata` module defines `SCHEMA_VERSION = 1`
- `Defaults.build` and `DefaultConfiguration.valid?` require `metadata` key
- `Validator` rejects configurations without `schema_version: 1`
- Migration **00026** migrates existing rows and updates column default

---

## 2. Configuration Metadata

Extended metadata structure:

```json
{
  "metadata": {
    "schema_version": 1,
    "last_updated_by": null,
    "updated_at": null
  }
}
```

On every PATCH (including reset):

- `last_updated_by` set to authenticated user id
- `updated_at` set to ISO8601 timestamp

Stored inside configuration JSON — no new database columns.

---

## 3. Renamed `tone.language` → `tone.default_language`

Represents preferred response language, not detected conversation language.

Updated: defaults, validator, controller params, migration, legacy normalizer, frontend types/schema/components, tests.

---

## 4. Renamed `greeting.message` → `greeting.default_greeting`

Represents fallback owner-configured greeting, not runtime-generated contextual greeting.

Updated: defaults, validator, controller params, migration, legacy normalizer, `GreetingEditor` component, frontend types/schema, tests.
